# 배치·소비 규율 — 사다리·관문·승격·회귀

> 계층이 있어도 배치·소비 규율이 없으면 구조는 장식이 된다. "어디서 태어나 언제 내려가는가"의 정본.

## 배치 사다리 (pages first — FSD 2.1 공식 + steiger insignificant-slice의 실행형)

```text
① 출생: 모든 새 코드는 라우트 콜로케이션에서 태어난다
   routes/<경로>/X.view.svelte (+필요시 X.live.svelte) — 접미사 의무, 무표 금지
② 하강 관문: 둘째 소비자(다른 라우트·다른 slice)가 등장하는 순간에만 내린다
   ├─ 업무 개체의 표현·타입·API (명사)          → entities/<개체>   (ui는 view 전용)
   ├─ 사용자가 "하는 일" (동사 — 폼·다이얼로그·액션) → features/<동사구>
   ├─ 여러 entity·feature의 자립 조립 블록          → widgets/<블록>
   └─ 업무 어휘 0 + 승격 4테스트 전부 통과          → shared/ui
③ 판정 불확실 → 높은 계층에 둔다 (widgets 디폴트) — 미리 나누는 것이 Atomic·FSD 2.0의 공통 사인
④ 회귀: 소비자가 라우트 1개로 줄면 콜로케이션으로 되돌린다 (arch:analyze INSIGNIFICANT_SLICE 신호)
```

- entity ui에 live를 만들고 싶다 = **widget 승격 신호** (자기 데이터를 무는 블록의 FSD 정의가 widget).
- 하강은 이동 + slice public API(index.ts) 개설 + 소비처 임포트 재작성을 **한 커밋**에.

## shared/ui 승격 관문 4테스트 (전부 통과해야 진입 — "shared/ui는 태어나지 않는다, 승격된다")

| # | 테스트 | 판정 질문 |
|---|---|---|
| T1 | 어휘 | 이름·Props·기본 문구·참조 타입에 제품/업무 단어 0인가? wire 타입 참조 = 즉시 탈락 |
| T2 | 이식 | 다른 회사 제품에 복사 + 디자인 토큰 매핑만으로 성립하는가? |
| T3 | 정본 | 정답 모양의 소유자가 스타일 가이드인가 화면 기획인가? 변경이 "가이드 개정"으로 오면 통과 |
| T4 | 수요 | 콜사이트 2+ (Rule of Two) 또는 가이드 명시 표준 부품인가? |

어휘 논쟁이 생기면(예: "프로필"이 업무어인가) 논쟁하지 말고 T3(정본 소유자)로 판정한다.

## 승격 절차 — 6단계 한 커밋

1. **역할명** — 업무를 지운 이름(`FooSearchBar` ✗ → `SearchInput` ✓). 안 지워지면 승격 대상이 아니라 entity/feature 부품.
2. **요구 매트릭스** — 기존 손구현 전 콜사이트의 (시각 × 인터랙션) 표. variant는 이 표에서만 — 미래 추측 금지.
3. **작성** — 템플릿 기반: 토큰만·업무 기본값 없이·a11y/IME 내장·@component+TSDoc·스토리/스펙.
4. **전 콜사이트 이관** — 같은 커밋에. "만들어두고 점진 이관"은 스타일 계열 +1일 뿐.
5. **재발방지 룰** — 그 패턴 손구현을 잡는 룰을 `.svelte-arch/config.mjs`에 같은 커밋으로. 룰 목록 = 승격의 역사.
6. **잔여 래퍼 정리** — 업무 문구만 남긴 얇은 view로 축소 or 소비처 1곳이면 인라인 후 삭제.

## 강등·회귀

- shared/ui에서 업무 지식 발견 → prop화 후 잔류, 구조 자체가 화면 전용이면 소속 slice로 이동(.view 유지 — git 이력에 사건으로).
- slice 소비자가 1로 줄면(analyze 신호 2릴리스 연속) 콜로케이션 회귀 검토. 대체된 고아는 즉시 삭제.

## 소비 규율 5조

1. **그리기 전 매니페스트** — `arch:manifest`(+`--slice`) 실행·읽기. "몰랐다"는 사유가 아니라 절차 위반.
2. **있으면 소비, 모자라면 variant** — ① shared/ui에 variant/prop 추가 ② 독립 역할이면 승격 절차 ③ 진짜 일회성만 이스케이프 해치.
3. **Rule of Two** — 같은 시각 패턴의 두 번째 구현 시점 = 승격 시점.
4. **해치는 일회성** — REPLACE형 class prop 계약: 설정 시 그 영역 스타일을 호출자가 전부 소유. 같은 해치값 2파일 = 위반. 클래스 문자열 상수 모듈로의 세탁도 동일 위반(`CLASS_CONST_EXPORT`).
5. **shared/ui에 업무 기본값 금지** — 문구성 prop은 기본값 삭제 → required → 호출부가 공급 책임자.

## 전형적 실패 모드 (규율이 없을 때 반드시 일어나는 일)

Input이 아이콘 슬롯까지 지원해도, 규율이 없으면 검색 인풋이 십수 곳에서 제각각 재조립된다 — raw input / 전용 래퍼 / 해치 복붙이 공존하고 "다른 화면과 통일" 주석을 수동 유지하게 된다. 어느 쪽이 의도된 디자인인지 아무도 모르는 드리프트 고고학. 매니페스트 주입 + 사다리 + 감사가 이걸 기계적으로 막는다.
